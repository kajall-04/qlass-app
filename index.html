Our Solution — How It Works

The platform operates through five integrated layers, from hospital setup to AI-powered decision support.

1

Hospital Onboarding

An admin creates a hospital workspace (e.g., "City General Hospital"), invites staff, and assigns roles: Owner, Admin, Department Head, Doctor, Nurse, Viewer. Each hospital's data is completely isolated — this is multi-tenancy.

2

Clinical Event Tracking

Every patient interaction becomes a tracked event — admissions, vital sign recordings, lab results, medications, transfers, discharges. Each event is timestamped and tagged with hospital ID, patient ID, event type, and rich metadata in JSON format.

3

AI Early Warning Engine

The platform continuously monitors patient vitals and history, calculating real-time risk scores for sepsis, cardiac events, patient deterioration, and readmission risk. Alerts fire instantly when thresholds are crossed.

4

Operations Dashboard

A live command center showing bed occupancy, emergency wait times, high-risk patient counts, lab turnaround, and staff workload — with interactive charts trending over time.

5

AI Clinical Assistant

Doctors ask questions in plain English — "Which patients are at highest risk right now?" — and the AI fetches relevant data, computes analytics, and returns actionable answers with recommended next steps.

Database Design

A normalized relational schema designed for multi-tenant isolation, clinical event tracking, and audit compliance.

hospitals
id UUID · PK

users

id UUID · PK

id UUID · PK
hospital_id FK → hospitals
patient_id FK → patients
user_id FK → users
event_type VARCHAR
timestamp TIMESTAMP
severity ENUM(normal/warning/critical)
metadata JSONB
alerts
id UUID · PK
patient_id FK → patients
alert_type VARCHAR
risk_score INTEGER (0–100)
status ENUM(active/acknowledged/resolved)
created_at TIMESTAMP
audit_logs
id UUID · PK
user_id FK → users
action VARCHAR
timestamp TIMESTAMP
details JSONB
Key Design Decisions: All tables reference hospital_id for tenant isolation. JSONB columns in events and audit logs provide flexible metadata storage without schema migrations. UUIDs are used as primary keys to prevent enumeration attacks and support distributed ID generation.

API Design & Endpoints

A RESTful API organized into four functional groups, all secured with JWT authentication and tenant-scoped data access.

🔐 Authentication & Hospital Management

POST /auth/register Create new user account
POST /auth/login Authenticate & receive JWT token
GET /me Get current user profile & roles
POST /hospitals Create new hospital workspace
GET /hospitals/:id Get hospital details & settings

🏥 Patient & Clinical Event APIs

POST /patients Admit new patient to hospital
GET /patients List all patients (filtered by tenant)
GET /patients/:id Patient detail + event timeline
POST /events/ingest Record clinical events (batch)
GET /events Query events with filters

📊 Analytics & Dashboard APIs

GET /analytics/bed-occupancy Bed usage breakdown by department
GET /analytics/high-risk-patients List patients with elevated risk scores
GET /analytics/sepsis-trend Sepsis alert volume over time
GET /analytics/wait-times Emergency wait time trends

🤖 AI & Alert APIs

POST /ai/ask Natural language clinical query
POST /ai/risk-score/:id Get AI risk prediction for patient
GET /alerts List active alerts (priority sorted)
POST /alerts/:id/acknowledge Mark alert as reviewed by clinician
